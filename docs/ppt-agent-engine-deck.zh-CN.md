## 01. 从斯坦福小镇开始：25 个 AI 角色自己生活起来

页面要点：

- 斯坦福团队在 2023 年提出 Generative Agents，在一个类似《模拟人生》的小镇里放入 25 个由大语言模型驱动的角色。
- 这些角色会醒来、吃饭、工作、交谈、记住经历，并根据记忆安排接下来的行动。
- 研究中最有代表性的例子是：只给出一个角色想办情人节派对的初始设定，邀请、传播、约会和集体赴约等行为逐步自然涌现。
- 这类系统展示了一个新方向：游戏世界里的角色不再只是脚本触发器，而可以成为有记忆、有目标、会互动的模拟个体。

讲解提示：

这不是简单的聊天机器人，也不是传统 NPC 脚本。关键变化在于，agent 能把看到的事存成记忆，再把记忆提炼成反思，最后用于计划下一步行为。斯坦福小镇让人第一次直观看到「AI 角色社会」的雏形。

画面建议：

- 一张俯视小镇/沙盒世界示意图。
- 叠加 3 个关键词：记忆、计划、社交。

## 02. 斯坦福小镇为什么重要

页面要点：

- 它把 LLM 从「回答问题」推进到「在环境中持续行动」。
- 它把单个 AI 对话推进到多个 agent 的长期交互。
- 它把预写剧情推进到可观察的涌现行为。
- 它证明了 agent 行为可信度不仅来自模型本身，也来自环境、记忆、规则和反馈循环。

讲解提示：

传统 NPC 的核心是脚本：玩家触发条件，系统播放预设反应。Generative Agents 的核心是状态循环：角色观察世界、更新记忆、做计划、执行动作，再影响世界。我们的项目正是沿着这个方向，把可编辑世界、规则引擎和本地模型接入做成一个可运行工具。

画面建议：

- 左侧：传统 NPC，分支脚本树。
- 右侧：Agent，观察 -> 记忆 -> 计划 -> 行动 -> 世界反馈闭环。

## 03. 从案例抽象出一个 agent 系统

页面要点：

- Agent 不是单次生成文本，而是一个持续运行的行为单元。
- 一个完整 agent 系统通常包含：世界状态、角色设定、记忆、计划、动作、规则校验和事件日志。
- 多 agent 系统还需要处理：相遇、对话、协作、冲突、资源争夺和群体行为。
- 对游戏和仿真来说，真正有价值的是「可控的涌现」，不是完全失控的自动生成。

讲解提示：

我们可以把 agent 理解成「模型 + 身体 + 环境 + 规则」。LLM 提供语言理解和决策能力，但它必须被放进一个有坐标、有物体、有约束、有日志的世界里，才会变成可观察、可调试、可复现的系统。

画面建议：

- 中心放一个 agent。
- 四周放：地图、物体、其他 agent、事件日志、规则引擎、模型服务。

## 04. 现有 agent 研究给我们的启发

页面要点：

- Generative Agents：强调长期记忆、反思和日程计划，让角色行为更可信。
- ReAct：把推理和行动交替起来，让模型边思考边调用工具或操作环境。
- Voyager：在 Minecraft 中通过环境反馈积累技能库，展示开放式探索和技能复用。
- AutoGen / CAMEL：把多个 LLM agent 组织成对话式协作系统。
- AgentBench：提醒我们 agent 需要在交互环境中评测，不能只看单轮问答。

讲解提示：

这些研究共同说明一件事：agent 的能力不是只靠更大的模型堆出来的。真正的工程问题是如何给模型安排观察、记忆、工具、动作空间、错误反馈和安全边界。

画面建议：

- 做成一张研究地图：记忆与仿真、推理与行动、技能学习、多 agent 协作、评测与安全。

## 05. Agent 技术正在从「聊天」走向「可执行」

页面要点：

- 第一阶段：用户和模型对话，模型主要输出文本。
- 第二阶段：模型能调用工具，例如搜索、读写文件、生成代码、控制浏览器。
- 第三阶段：模型作为 agent 进入环境，拥有状态、目标和行动循环。
- 第四阶段：多个 agent 在同一环境中交互，形成任务协作或社会仿真。

讲解提示：

我们的项目瞄准第三和第四阶段：不是再做一个聊天框，而是给 agent 一个可以行动的世界。这个世界可视化、可编辑、可追踪，并且尽量本地优先。

画面建议：

- 阶梯图：Chatbot -> Tool User -> Single Agent -> Multi-Agent World。

## 06. Multi-Agent AI Game Engine 的定位

页面要点：

- 它是一个本地优先的多 agent 场景编辑器和模拟器。
- 用户可以搭建 2D 世界：地图、区域、物体、角色、规则和事件。
- Agent 可以移动、观察、说话、社交、拾取物体，并由本地或远程 LLM 驱动。
- 系统保留规则引擎和事件日志，让模型行为可以被约束和追踪。

讲解提示：

这个工具不是传统游戏引擎的替代品，而是一个「可运行的世界编辑器」。它的重点不是渲染多华丽，而是让用户能搭一个有规则的世界，然后观察 agent 在里面怎么行动。

画面建议：

- 项目界面截图。
- 角标标注：地图、agent、item、模型管理、事件日志。

## 07. 为什么要做成本地优先

页面要点：

- 场景数据、角色设定、地图素材和事件日志默认保存在本机。
- Windows 桌面版内置后端，普通用户不需要安装 Python。
- 本地模型可以通过 Ollama 等方式接入，适合离线演示和隐私敏感场景。
- 数据目录和安装目录分离，方便升级应用，也方便备份项目。

讲解提示：

多 agent 世界会产生大量私有上下文：角色身份、地图、对话、行为记录、模型配置。把这些数据默认留在本机，会让试验、演示和二次开发都更稳。

画面建议：

- 一张本地架构图：Electron 前端、本地后端、SQLite、assets、Ollama。

## 08. 用户拿到应用后会看到什么

页面要点：

- Windows 安装版：`Multi-Agent Engine-0.1.0-win-installer-x64.exe`。
- Windows 便携版：`Multi-Agent Engine-0.1.0-win-portable-x64.exe`。
- 安装版会把应用安装到系统用户目录，便携版可以直接双击运行。
- 当前安装包包含前端、Electron、后端可执行文件和用户手册；不内置大型模型权重。

讲解提示：

这里要把边界讲清楚：应用本体已经可以独立运行，但 LLM 模型、Ollama 模型和 SAM 大模型不建议塞进主安装包。模型更适合做成可选离线包或安装后的一键配置流程。

## 09. 第一次启动：从空世界到可运行场景

页面要点：

- 打开应用后，系统会创建或读取本机 `runtime_project`。
- 用户可以从默认沙盒开始，也可以导入已有项目数据。
- 第一步建议先确认后端连接、地图显示和事件日志可用。
- 初次体验可以先用 mock provider，不必马上接真实 LLM。

讲解提示：

新用户最容易被模型配置卡住，所以建议演示时先让世界跑起来，再接模型。只要地图、agent、事件日志和单步模拟正常，用户就能理解这个工具的核心价值。

画面建议：

- 启动后的主界面。
- 用箭头标注「先跑通模拟，再接模型」。

## 10. 工作台：把世界变成可编辑对象

页面要点：

- 中央工作台用于查看地图、选择对象、绘制区域和移动视角。
- 浮动面板用于管理地图、agent、item、模型、事件和叙事。
- 用户可以缩放、平移、选中对象，并查看对象属性。
- 所有对象最终都会进入世界状态，被模拟器和规则引擎使用。

讲解提示：

工作台的价值是把抽象的 agent 系统变成可操作的空间。用户不是写一堆 JSON，而是在地图上直接搭建 agent 的生活环境。

画面建议：

- 主工作台截图。

## 11. 地图：agent 行动的空间基础

页面要点：

- 地图可以是导入图片，也可以通过生成流程得到候选图。
- 地图不仅是背景图，还承载坐标、区域、物体和出生点。
- 小镇、房间、基地、校园、城市街区都适合做成 2D 场景。
- 建议先做小场景，确认行为闭环后再扩大地图。

讲解提示：

对 agent 来说，地图不是装饰。地图决定它在哪里、能去哪里、会遇到谁、能和什么物体互动。地图越清楚，后续 agent 行为越容易调试。

画面建议：

- 一张俯视小镇地图或项目示例地图。

---

## 12. 区域：给地图赋予语义

页面要点：

- 区域用于标注道路、障碍、行动区、居住区和社交区。
- 手动绘制区域适合精细控制，SAM 分层适合快速切出候选形状。
- 区域语义会影响 agent 的移动、交互和规则校验。
- 好的区域命名可以帮助 LLM 理解当前环境。

讲解提示：

同一张地图，如果没有区域语义，模型只看到一张图或一堆坐标；有了区域，模型和规则引擎才知道哪里是路、哪里不能走、哪里适合社交。

画面建议：

- 地图上叠加半透明区域色块。
- 做一个图例：道路、障碍、社交区、居住区。

## 13. Agent：世界中的行动者

页面要点：

- 每个 agent 包含身份、角色、目标、当前位置、动作空间和模型配置。
- Agent 可以执行移动、说话、观察、等待、社交、拾取和移动物体等动作。
- 角色设定越清楚，模型越容易生成稳定行为。
- 多个 agent 的相遇、对话和冲突会形成可观察的场景动态。

讲解提示：

创建 agent 时，不要只给名字。最好说明它是谁、关心什么、现在想做什么、它能执行哪些动作。这样它才像一个角色，而不是随机移动的点。

画面建议：

- Agent 面板截图。
- 标注 profile、state、actions、model。

## 14. Item：让世界可以被操作

页面要点：

- Item 是地图上的物体，例如箱子、门、家具、资源、标记物。
- Item 可以是纯视觉对象，也可以参与 agent 动作。
- 可移动 item 可以被拾取、放下或移动；不可移动 item 可作为设施、障碍或地标。
- Item 和 action extension 配合后，可以扩展出修理、开门、摧毁、交换等行为。

讲解提示：

没有物体的世界很难产生有意义的行为。Item 是让 agent 从「走来走去」进入「做事情」的关键。

画面建议：

- 地图上的物体示意。
- 用不同图标区分 movable 和 static item。

## 15. 模拟：用 tick 推进世界

页面要点：

- 系统按 tick 推进模拟，每个 tick 都可能产生移动、对话、动作和事件。
- 用户可以播放、暂停、单步推进，也可以停止单个 agent。
- 调试时建议用单步，演示时再使用播放。
- 所有关键变化都会进入事件日志，方便复盘。

讲解提示：

tick 的好处是把连续世界切成可观察的小步。用户可以看到某一步模型做了什么、规则接受了什么、世界状态如何变化。

画面建议：

- 播放控制条截图。
- 配合事件日志截图。

## 16. LLM 接入：让 agent 的行为不再固定

页面要点：

- 系统支持 mock provider、本地 Ollama、远程兼容服务等配置思路。
- LLM 可用于 agent 决策、对话、地图/区域理解和叙事生成。
- 模型不是直接改世界，而是提出动作请求，再由规则引擎校验。
- 小模型适合快速本地试验，大模型适合更复杂的推理和叙事。

讲解提示：

这里最重要的是「模型提出，系统校验」。这样既能利用 LLM 的灵活性，又能避免它随意破坏世界状态。

画面建议：

- 模型管理面板。
- 流程图：LLM response -> parse action -> validate -> apply -> event。

## 17. 动作扩展：把用户想象力接进规则系统

页面要点：

- 默认动作覆盖移动、说话、观察、等待、社交和物体操作。
- 动作扩展允许用户新增受控动作，例如修理设施、开门、摧毁物体、触发剧情。
- 每个扩展动作需要声明参数、校验逻辑、执行逻辑和权限。
- LLM 可以请求动作，但动作是否生效由系统验证。

讲解提示：

动作扩展是从「演示工具」走向「可定制引擎」的重要一步。用户可以把自己的玩法、实验规则或业务逻辑变成 agent 可使用的动作。

画面建议：

- 展示一个 `repair_item` 或 `open_door` 的简单动作卡片。

---

## 18. 叙事系统：给世界一个持续的故事语境

页面要点：

- 叙事面板用于配置场景前提、语气、节奏和自动旁白。
- 它可以把零散事件组织成更容易理解的故事流。
- 适合演示、原型验证、教育训练和游戏剧情实验。
- 叙事不应替代事件日志；它更像给观察者看的解释层。

讲解提示：

事件日志回答「发生了什么」，叙事层回答「这些事看起来像什么故事」。两者配合，才能兼顾可审计和可展示。

画面建议：

- 一侧事件日志，一侧旁白文字。

## 19. 可审计性：为什么事件日志很重要

页面要点：

- 普通事件记录世界发生了什么。
- Decision events 记录模型为什么提出某个动作、动作如何被解析、是否通过校验。
- 调试 agent 行为时，不能只看画面，还要看决策链。
- 日志让系统更适合研究、演示、教学和问题复盘。

讲解提示：

Agent 系统最怕「看起来很智能，但不知道为什么」。事件日志和 decision events 是把黑盒行为拆开的工具，也是后续评测和优化的基础。

画面建议：

- 事件详情卡片。
- 标出 raw response、parsed action、validation result。

## 20. 推荐演示流程：10 分钟跑通一个小镇

页面要点：

- 第 1 步：打开安装版或便携版，确认后端已连接。
- 第 2 步：载入或导入一张小镇地图。
- 第 3 步：绘制道路、广场、住宅区和障碍区域。
- 第 4 步：创建 3 个 agent，分别设置身份和目标。
- 第 5 步：放置 2-3 个 item，例如公告板、箱子、门。
- 第 6 步：单步运行，观察移动、对话和事件日志。
- 第 7 步：切换真实 LLM 或演示 mock provider 的稳定流程。

讲解提示：

演示时不要一开始就追求大地图和大量角色。三五个 agent 已经足够展示多 agent 世界的核心：它们会在同一空间中行动、相遇、互动，并留下可追踪记录。

画面建议：

- 做成流程时间轴。

## 21. 面向游戏的价值

页面要点：

- 更像人的 NPC：角色可以记住事件、形成日程、主动对话。
- 更低成本的原型：策划可以快速验证场景规则和角色互动。
- 更灵活的剧情：从固定脚本扩展到半开放事件链。
- 更好的调试：事件日志帮助设计者理解角色行为来源。

讲解提示：

这套系统并不是要取代成熟游戏引擎，而是补足「角色行为原型」和「叙事仿真」这一层。它可以先验证玩法和世界规则，再决定如何接入正式游戏管线。

画面建议：

- NPC 对话、巡逻、聚会、任务触发的组合示意。

## 22. 面向研究、教育和训练的价值

页面要点：

- 研究：观察多 agent 行为、社会互动和规则变化的影响。
- 教育：用可视化方式讲解 AI agent、仿真和复杂系统。
- 训练：构建谈判、应急、服务、团队协作等可演练场景。
- 原型：快速测试「如果多个角色都按各自目标行动，会发生什么」。

讲解提示：

Agent 世界的价值不仅在游戏。它也可以作为「情境模拟器」：把人物、环境和规则放进去，看系统如何演化。

画面建议：

- 四象限：游戏、教育、研究、训练。

## 23. Agent 使用前景：从工具到可运行社会

页面要点：

- 短期：个人和小团队用 agent 自动化任务、辅助创作、生成原型。
- 中期：企业把多个 agent 组织成流程协作，处理研发、运营、数据分析和客服场景。
- 长期：agent 与仿真环境结合，形成城市、社区、组织和市场的行为实验平台。
- 关键限制仍然存在：可靠性、评测、隐私、成本、模型偏差和可控性。

讲解提示：

最值得期待的不是「一个万能 agent」，而是一组有边界、有工具、有记忆、能被审计的 agent。多 agent 世界会成为研究复杂行为、游戏叙事和业务流程自动化的共同底座。

画面建议：

- 三段路线图：现在、下一阶段、长期。

## 24. 当前边界：哪些东西还不应该过度承诺

页面要点：

- 当前版本是可运行的本地工具，不是完整商业游戏引擎。
- 安装包不随附大型本地模型；模型更适合通过一键配置或独立模型包安装。
- Agent 行为依赖模型质量、角色设定、区域语义和规则约束。
- 自动生成结果需要人工审核，尤其是地图分层、动作扩展和叙事输出。
- 对外发布前还需要补充代码签名、版本管理和更完整的错误提示。

讲解提示：

这页很重要，因为它能让介绍更可信。我们不把它包装成万能 AI，而是明确说：这是一个把 agent 放进可编辑世界里的工程底座。

画面建议：

- 左侧「已可用」，右侧「待增强」。

## 25. 建议用户这样开始

页面要点：

- 先用便携版或安装版打开应用，确认基础界面和后端连接正常。
- 先使用 mock provider 跑通地图、agent、item 和事件日志。
- 再接入 Ollama 或其他 LLM，观察真实模型行为。
- 先做小世界，再扩成复杂场景。
- 每次重要实验前备份 `runtime_project`。

讲解提示：

新用户最稳的路线是「小场景、少 agent、先规则、后模型」。先让世界能跑，再追求智能行为。

画面建议：

- 检查清单式页面。

## 26. 我们最终想交付什么

页面要点：

- 一个普通用户能打开的 Windows 桌面应用。
- 一个开发者能继续扩展的本地多 agent 引擎。
- 一个能连接 LLM、地图、规则、事件和叙事的可运行世界。
- 一个从斯坦福小镇研究思路出发、落到实际工具链的 agent 实验平台。

讲解提示：

这套项目的核心不是「让模型说得更像人」，而是「让模型进入一个可运行、可约束、可观察的世界」。当角色、物体、区域、规则和模型都在同一个系统里，agent 才真正从聊天走向行动。

画面建议：

- 项目 logo 或主界面大图。
- 结尾标语：Build worlds. Run agents. Observe emergence.

# 附录 C：一句话讲清楚这个项目

Multi-Agent AI Game Engine 是一个本地优先的可视化 agent 世界编辑器：用户可以搭建地图、区域、物体和角色，让多个由规则和 LLM 共同驱动的 agent 在同一个世界中移动、对话、互动，并通过事件日志观察它们的行为如何涌现。

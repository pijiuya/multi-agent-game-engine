# Multi-Agent AI Game Engine 用户手册

本文面向使用者，而不是开发者。你可以把它当作从安装、第一次启动、创建场景、配置模型、运行模拟到排障的完整操作说明。

> 当前版本是本地优先的多 Agent 场景编辑器和模拟器。它可以在网页开发模式中运行，也可以被打包成 Windows 桌面应用。应用会把项目数据保存在本机，不会自动上传地图、角色、事件或模型配置。

## 1. 你可以用它做什么

Multi-Agent AI Game Engine 用来搭建一个可视化世界，并让多个 agent 在这个世界中行动、对话、互动和被模型驱动。

典型用途包括：

- 设计一个 2D 场景地图，例如小镇、房间、基地、校园、迷宫、城市街区。
- 在地图上绘制道路、障碍、行动区、居住区、社交区等功能区域。
- 创建多个 agent，设置身份、性格、目标、动作空间、对话策略和动画。
- 放置物体 item，并设置它们是否可移动、是否可被拾取、是否隐藏。
- 启动模拟，让 agent 移动、停止、观察、说话、社交、拾取和移动物体。
- 接入本地 LLM，让 agent 的行为由模型生成，同时保留规则引擎校验。
- 使用地图生成、图片导入、SAM 分层、图像识别等能力辅助建图。
- 使用事件日志和 decision events 追踪模型做了什么、为什么被接受或拒绝。
- 使用动作扩展为 agent 增加自定义动作，例如修理、摧毁、开门、触发剧情。
- 使用叙事面板维护场景前提、语气、节奏和自动旁白。

这个工具不是传统游戏引擎的替代品。它更像一个“可运行的世界编辑器”：先把世界和规则搭起来，再观察 agent 在其中如何行动。

## 2. 版本形态

### 2.1 Windows 桌面版

如果你拿到的是 Windows 打包产物，通常会看到两类文件：

- `Multi-Agent Engine-0.1.0-win-installer-x64.exe`：安装器，会把应用安装到系统目录。
- `Multi-Agent Engine-0.1.0-win-portable-x64.exe`：便携版，直接双击运行，不需要安装。

两者功能相同。建议普通用户优先使用 installer；测试、演示或临时分发时可以使用 portable。

桌面版内置后端服务。正常情况下你不需要安装 Python，也不需要手动启动后端。

### 2.2 开发版

开发版需要分别启动后端和前端：

```powershell
# 仓库根目录
$env:AGENT_ENGINE_PROJECT_DIR = "runtime_project"
.\.venv\Scripts\python -m uvicorn agent_engine.api.main:app --app-dir backend --host 127.0.0.1 --port 8000
```

另开一个终端：

```powershell
cd frontend
npm run dev -- --port 5173
```

然后打开：

```text
http://127.0.0.1:5173/
```

开发版适合调试、二次开发和查看日志。普通使用不需要这套流程。

## 3. 数据保存在哪里

### 3.1 桌面版数据目录

Windows 打包版默认把项目数据保存在用户数据目录下：

```text
%APPDATA%\Multi-Agent Engine\runtime_project\
```

这里通常包含：

- `world.sqlite`：主要项目数据库，保存地图、agent、item、区域、事件、模型配置等。
- `project.json`：项目元数据。
- `assets/`：上传或生成的地图、物体图片、agent 动画素材。
- `models/`：本地模型缓存，例如 MobileSAM 权重。
- `logs/`：打包版后端日志。

如果要迁移桌面版项目，优先备份整个 `runtime_project` 目录。

### 3.2 开发版数据目录

开发版默认使用仓库根目录下的：

```text
runtime_project/
```

这个目录通常不会提交到 Git。它是你的本机项目状态。

### 3.3 数据安全建议

- 做重要实验前，复制一份 `runtime_project` 作为备份。
- 不要只备份截图；真正的世界状态在 `world.sqlite` 中。
- 不要把包含私人 API key、模型地址、场景数据的 `runtime_project` 直接发给不可信的人。
- 如果你要提交代码，确认 `runtime_project/`、`frontend/release/`、`packaging/backend-dist/` 等构建或运行目录没有被提交。

## 4. 第一次启动

### 4.1 使用 Windows installer

1. 双击 `Multi-Agent Engine-0.1.0-win-installer-x64.exe`。
2. 如果 Windows 提示未知发布者，这是因为当前版本未做正式代码签名。
3. 选择安装目录。
4. 安装完成后启动 `Multi-Agent Engine`。
5. 进入主界面后确认顶部或状态栏显示后端已连接。

### 4.2 使用 portable

1. 双击 `Multi-Agent Engine-0.1.0-win-portable-x64.exe`。
2. 等待窗口打开。
3. 第一次启动时后端需要初始化 SQLite 数据库，可能比之后慢一点。
4. 如果窗口出现但数据显示为空，可以等待几秒或重新打开。

### 4.3 首次启动后的检查

启动后你应该能看到：

- 一个透明风格的 2D 工作台。
- 地图或默认沙盒。
- 场景列表、工具、属性、模型管理等浮动面板。
- 若已有项目数据，则会看到旧的 agent、地图、事件和区域。
- 若是新项目，则会看到默认 agent 和默认场景。

如果你看到“后端未连接”或界面回退到默认数据，请看本文的“排障”章节。

## 5. 主界面概览

主界面由几个部分组成：

- 工作台：中间的大区域，用于查看地图、移动视角、选择对象、绘制区域和编辑物体。
- 浮动面板：可拖动、吸附、折叠和调整大小的工具窗口。
- 场景列表：展示地图、agent、item、出生点、区域等对象。
- 工具面板：切换选择、绘制、锚点、区域编辑等操作模式。
- 属性面板：编辑当前选中对象的名称、位置、尺寸、显示状态和行为配置。
- Agent 面板：管理 agent 的动作、动画、对话策略、附近对象和控制按钮。
- 模型管理：检测并配置 LLM、图片生成和 SAM 分层能力。
- 地图工坊：导入或生成地图，运行分层和区域识别。
- 事件日志：查看普通 world events 和模型决策事件。
- 叙事面板：配置场景前提、语气、节奏和自动旁白。

不同版本的面板名称可能略有变化，但核心功能大体一致。

## 6. 工作台操作

### 6.1 选择对象

选择工具用于点击地图上的 agent、item、区域和锚点。

常见操作：

- 单击对象：选中对象，并在属性面板显示细节。
- 双击 agent：将视角居中到该 agent。
- 右键对象：打开上下文菜单，执行隐藏、显示、删除、重命名等操作。
- 点击空白处：取消或改变当前选择。

### 6.2 缩放和平移

常见操作：

- 鼠标滚轮：缩放工作台。
- 中键拖动：平移无限网格。
- 原点按钮：将视角回到世界坐标原点。
- 100% 按钮或缩放显示：回到默认缩放。

建议先掌握缩放和平移，再进行精细绘制。

### 6.3 网格和坐标

工作台使用世界坐标。agent、item、区域点位都保存为地图坐标。

注意：

- 缩放只改变视图，不会改变对象真实坐标。
- 平移只移动视角，不会移动地图或对象。
- 绘制区域时，点位会落在当前地图坐标上。
- 如果对象位置异常，先确认当前缩放和平移状态。

### 6.4 浮动面板

浮动面板用于组织复杂操作。

常见操作：

- 拖动标题栏：移动面板。
- 靠近边缘：面板可吸附到场景边缘。
- 点击最小化按钮：折叠或展开面板。
- 拖动边缘：调整面板大小。
- 重新打开页面后：面板布局通常会尝试保留。

如果面板挡住工作台，可以拖到边缘或折叠。

## 7. 地图管理

### 7.1 地图是什么

地图是当前世界的空间基础。它包含：

- 宽度和高度。
- 背景图片。
- 区域 regions。
- 渲染层 region layers。
- item。
- 出生点或锚点。

地图不是单纯图片。背景图只是视觉底图，真正影响 agent 行为的是区域、障碍、规则和对象。

### 7.2 导入地图图片

在地图工坊或地图相关面板中选择导入图片。

建议：

- 使用清晰的俯视图。
- 避免过大的原图，尤其是演示机器性能有限时。
- 如果地图有明显道路、房屋、障碍，后续 SAM 分层会更容易。
- 导入后确认地图宽高和显示比例。

导入图片会保存到项目的 `assets/` 目录，并在数据库中记录引用。

### 7.3 生成地图

地图生成通常会根据文本提示生成候选图。

基本流程：

1. 打开地图工坊。
2. 输入地图提示，例如“俯视小镇地图，有道路、广场、住宅区和树林”。
3. 设置比例和尺寸。
4. 点击生成。
5. 从候选图中选择一个作为当前地图背景。

如果没有配置真实图片生成模型，系统可能使用 mock 候选图或提示你配置本地/远程服务。

### 7.4 地图尺寸

地图尺寸会影响坐标系统和对象位置。修改尺寸前要注意：

- 已存在对象的坐标不会自动按比例重排，除非界面提供相关操作。
- 缩小地图可能让对象看起来跑到边界外。
- 大地图会带来更多绘制和移动空间，但也更难管理。

建议在搭建初期确定大致尺寸。

## 8. 区域系统

### 8.1 区域的作用

区域用于给地图赋予功能语义。

常见区域类型：

- 道路：适合 agent 移动。
- 障碍：agent 不应穿过。
- 行动区：允许特定行为或目标。
- 居住区：适合角色停留、休息或生活。
- 社交区：适合对话和集会。
- 未分类区域：暂时保留，后续再命名或分配功能。

规则引擎会根据区域判断某些动作是否合理。

### 8.2 手动绘制区域

基本流程：

1. 选择区域绘制工具。
2. 选择区域功能，例如道路或障碍。
3. 在地图上逐点点击形成多边形。
4. 完成绘制后保存区域。
5. 在区域面板中检查名称、功能、轮廓和可见性。

绘制建议：

- 多边形不要自交。
- 点位不要过密。
- 障碍区域应覆盖真实不可通行区域。
- 道路区域应形成连续移动路径。

### 8.3 布尔编辑

区域布尔编辑用于增加或扣减区域形状。

常见用途：

- 给道路补一块缺失区域。
- 从行动区里扣掉障碍。
- 合并多个相邻社交区。
- 修正 SAM 自动分层产生的边界。

如果布尔结果不符合预期，建议撤回或重新绘制更简单的多边形。

### 8.4 SAM 分层

SAM 分层会尝试把地图背景切分成多个区域。

使用前提：

- 已导入或选择地图背景。
- 已配置本地 SAM 或远程分割服务。
- 地图图片清晰，区域边界相对明显。

基本流程：

1. 打开模型管理。
2. 检查 SAM 分层能力。
3. 若未配置，使用一键本地安装或远程配置。
4. 回到地图工坊。
5. 运行分层。
6. 检查生成区域，删除无用区域，重命名有用区域。
7. 将区域功能设置为道路、障碍、居住区等。

注意：

- SAM 只负责“切出形状”，不一定理解语义。
- 自动命名依赖图像识别或 LLM 能力。
- 分层结果通常需要人工整理。

### 8.5 区域命名和自动识别

区域可以手动命名，也可以通过模型自动命名。

建议命名方式：

- `主街道`
- `北侧住宅区`
- `中心广场`
- `树林障碍`
- `商店入口`

好的命名会让 LLM 更容易理解 agent 所在环境。

## 9. Agent 管理

### 9.1 Agent 是什么

Agent 是世界中的行动者。它包含两部分：

- Profile：身份、名字、角色、性格、目标、动作空间、模型配置等。
- State：当前位置、目标点、状态、持有物体、叙事状态等。

Profile 更像角色设定，State 更像运行时状态。

### 9.2 创建 agent

常见方式：

- 在锚点或出生点上生成 agent。
- 在场景列表或 agent 面板中新增。
- 通过 API 或导入数据创建。

创建后建议立即设置：

- 名称。
- 角色。
- 性格或身份描述。
- 初始位置。
- 是否隐藏。
- 模型 provider。
- 可执行动作。

### 9.3 编辑 agent 基础信息

在属性面板或 agent 面板中可以编辑：

- 名称：显示在场景和列表中。
- 角色：用于描述 agent 的社会定位。
- 身份描述：提供给 LLM 的背景信息。
- 目标：影响 LLM 决策。
- 位置：当前坐标。
- 显示状态：隐藏或显示。

建议给 agent 写清楚“它是谁”和“它想做什么”。这比只给名字更有用。

### 9.4 Agent 动作

内置动作通常包括：

- `move_to`：移动到目标点。
- `say`：说话。
- `observe`：观察。
- `wait`：等待。
- `interact`：互动。
- `use`：使用对象。
- `stop`：停止或休息。
- `social`：与其他 agent 社交。
- `pick_up`：拾取可移动 item。
- `drop_item`：放下 item。
- `move_item`：移动持有或附近 item。

如果配置了动作扩展，agent 还可以获得自定义动作。

### 9.5 对话策略

对话策略用于控制 agent 是否自动社交。

常见参数：

- 是否启用自动对话。
- 对话距离。
- 冷却 tick。

建议：

- 人群密集场景可以设置较短冷却，产生更多互动。
- 大地图探索场景可以设置较长冷却，避免刷屏。
- 对话距离不要过大，否则远距离 agent 也会频繁对话。

### 9.6 Agent 动画

Agent 可以绑定 GIF 或 PNG 序列帧。

可配置内容：

- 动画类型。
- 帧率 FPS。
- 最大像素数。
- 显示缩放。

如果动画过大，可能导致界面卡顿。建议先使用小尺寸素材验证效果。

## 10. Item 管理

### 10.1 Item 是什么

Item 是地图上的物体，可以是：

- 家具。
- 箱子。
- 门。
- 资源。
- 标记物。
- 可移动道具。

Item 可以只作为视觉对象，也可以参与 agent 动作。

### 10.2 可移动与不可移动

`movable` 决定 item 是否能被 agent 拾取或移动。

- 可移动 item：agent 可以拾取、放下、移动。
- 不可移动 item：通常作为障碍、建筑、地标或固定设施。

如果一个对象是墙、门框、大树、建筑，建议设为不可移动。

### 10.3 Item 图片

Item 可以绑定图片资源。

建议：

- 使用透明背景 PNG。
- 控制图片尺寸，避免过大。
- 给不同类别物体使用明显视觉差异。

### 10.4 Item 交互距离

Agent 与 item 互动通常需要距离足够近。

如果 agent 无法拾取或操作 item，检查：

- item 是否隐藏。
- item 是否 movable。
- agent 是否在互动距离内。
- agent 是否已持有其他 item。
- 动作 payload 是否指定了正确 item id。

## 11. 模拟控制

### 11.1 Tick

模拟按 tick 推进。每个 tick 可能发生：

- agent 移动一步。
- LLM 生成动作。
- 规则引擎校验动作。
- item 跟随持有者。
- 写入事件日志。
- 更新 decision events。
- 生成叙事或环境事件。

### 11.2 播放、暂停和单步

常见控制：

- 播放：持续推进模拟。
- 暂停：停止自动推进。
- 单步：只推进一个 tick。
- 停止 agent：让某个 agent 停止当前动作。

调试时建议使用单步。演示时使用播放。

### 11.3 事件日志

事件日志用于回答“刚刚发生了什么”。

常见事件类型：

- agent 移动。
- agent 说话。
- agent 社交。
- item 被拾取或移动。
- 环境事件。
- 叙事事件。
- 模型决策事件。

如果模拟行为不符合预期，先看事件日志，再看 agent 配置。

### 11.4 Decision events

Decision events 用于审计模型决策。

它通常会记录：

- 哪个 agent 请求了模型。
- 使用了哪个 provider。
- 模型原始响应。
- 解析出的动作。
- 动作是否通过规则校验。
- 最终关联到哪个 world event。

这对调试 LLM 行为非常重要。不要只看 agent 表面动作，也要看决策链。

## 12. 模型管理

### 12.1 能力分类

模型管理按能力划分：

- LLM：控制 agent 语言、意图、决策。
- 图片生成：生成地图背景或候选图片。
- SAM 分层：把地图图片切分成区域。
- 图像识别：给区域、物体或地图内容打标签。

不同能力可以由不同模型提供。

### 12.2 Mock provider

Mock provider 是内置测试提供者。

适合：

- 没有安装本地模型时快速体验。
- 开发和测试。
- 验证 UI 流程。
- 避免真实模型不稳定影响调试。

Mock provider 不代表真实智能行为。

### 12.3 使用 Ollama LLM

推荐流程：

1. 安装 Ollama。
2. 启动 Ollama 服务。
3. 拉取模型，例如：

```powershell
ollama pull qwen2.5:1.5b
ollama pull qwen2.5:7b
ollama pull gemma3:1b
```

4. 打开模型管理。
5. 点击重新检测。
6. 使用一键本地 LLM。
7. 启动模拟，观察 agent 行为。

如果模型响应慢，优先尝试小模型。

### 12.4 一键配置

一键配置会尝试检测本机服务，并生成合适的模型配置。

可能涉及：

- 检测 Ollama。
- 安装或启动本地能力。
- 拉取推荐模型。
- 写入模型配置。

注意：

- 一键安装可能需要网络。
- 模型下载可能很大。
- 如果安装失败，可以手动配置远程服务。

### 12.5 远程或兼容服务

高级配置通常需要：

- Base URL。
- API key。
- Model name。
- Provider 类型。
- 能力列表。

填写前确认服务兼容格式。不要把 API key 填到不可信服务。

## 13. 动作扩展

动作扩展用于新增内置动作之外的能力。

典型例子：

- 摧毁附近 item。
- 修理设施。
- 打开门。
- 给另一个 agent 物品。
- 改变区域状态。
- 触发剧情事件。

扩展通常包含：

- 动作元数据。
- 参数 schema。
- validate 函数。
- apply 函数。
- 权限声明。

安全原则：

- validate 只检查，不修改世界。
- apply 只在 validate 通过后执行。
- 高风险动作应明确权限。
- 不要让 LLM 直接随意修改世界。

更详细的编写说明见：

```text
docs/action-extension-manual.zh-CN.md
```

## 14. 叙事系统

叙事系统用于给场景增加连续故事感。

可配置内容：

- 场景前提：世界背景和当前冲突。
- 语气：例如冷静、童话、悬疑、写实、轻松。
- 节奏：多少 tick 生成一次叙事。
- 是否启用叙事。

叙事可能写入：

- 字幕。
- 场景记忆。
- agent 叙事状态。
- 环境事件。

建议：

- 前提不要太长，但要明确。
- 语气用短语描述。
- 节奏不要太密，否则事件日志会被叙事刷屏。

## 15. 常见工作流

### 15.1 从零搭一个小镇

1. 新建或打开沙盒。
2. 导入或生成小镇俯视地图。
3. 绘制道路区域。
4. 绘制住宅区、商店区、广场、树林障碍。
5. 放置几个 item，例如箱子、路灯、公告板、门。
6. 创建 3 到 5 个 agent。
7. 给 agent 写角色和目标。
8. 配置 LLM 或使用 mock provider。
9. 单步运行 10 个 tick，检查移动和事件。
10. 修正区域、位置和动作空间。
11. 播放模拟并观察日志。

### 15.2 调试 agent 不移动

检查顺序：

1. 是否暂停了模拟。
2. agent 是否隐藏。
3. agent 是否有移动动作。
4. 目标点是否在可行走区域。
5. 目标点是否落在障碍内。
6. LLM 是否返回了可解析动作。
7. 规则引擎是否拒绝了动作。
8. decision events 是否有错误原因。

### 15.3 调试 agent 不说话

检查顺序：

1. 是否启用了 LLM 或 mock provider。
2. agent 是否有 `say` 或 `social` 动作。
3. 对话策略是否启用。
4. 附近是否有其他未隐藏 agent。
5. 距离是否在对话范围内。
6. 冷却 tick 是否还没结束。
7. 模型响应是否为空或格式错误。

### 15.4 调试 item 无法拾取

检查顺序：

1. item 是否存在。
2. item 是否隐藏。
3. item 是否 `movable=true`。
4. agent 是否足够近。
5. agent 是否已经持有其他 item。
6. 动作 payload 是否带了正确 item id。
7. 规则引擎返回了什么错误。

### 15.5 准备一个演示项目

1. 使用尺寸适中的地图。
2. 只保留关键区域，避免区域太碎。
3. 创建少量但设定明确的 agent。
4. 给每个 agent 一个清晰目标。
5. 使用 mock provider 先确认流程。
6. 再切换到真实 LLM。
7. 备份 `runtime_project`。
8. 用 portable 包演示，避免安装步骤打断现场。

## 16. 打包版使用说明

### 16.1 Installer

Installer 适合长期使用。

特点：

- 会安装到系统应用目录。
- 可创建快捷方式。
- 卸载更方便。
- 数据仍保存在用户数据目录，不会直接放在安装目录。

### 16.2 Portable

Portable 适合临时运行。

特点：

- 双击即用。
- 不需要安装。
- 适合演示和快速分发。
- 仍会在用户数据目录写入项目数据和日志。

### 16.3 未签名提示

当前版本没有正式代码签名。Windows 可能显示：

- 未知发布者。
- Windows Defender SmartScreen 提示。
- 需要手动确认运行。

正式对外发布前建议补充代码签名。

### 16.4 本地模型不随包分发

打包版不包含大模型权重、Ollama 模型、MobileSAM 权重。

原因：

- 模型文件体积很大。
- 不同用户机器和 GPU/CPU 环境不同。
- 模型许可证可能不同。

用户需要在模型管理面板中重新检测或安装本地能力。

## 17. 排障

### 17.1 应用打不开

可能原因：

- Windows 阻止未知发布者。
- 杀毒软件拦截。
- 文件下载不完整。
- release 文件被移动后缺少依赖。

处理：

- 使用 portable 再试一次。
- 检查文件大小是否完整。
- 查看 Windows 安全中心拦截记录。
- 从可信来源重新复制完整 release 目录或 exe。

### 17.2 界面打开但后端未连接

处理：

- 等待 5 到 10 秒，首次启动可能初始化较慢。
- 关闭应用后重新打开。
- 检查是否已有其他程序占用 8000 端口。
- 查看日志：

```text
%APPDATA%\Multi-Agent Engine\logs\backend.err.log
%APPDATA%\Multi-Agent Engine\logs\backend.out.log
```

开发版则检查后端终端输出。

### 17.3 模型不可用

检查：

- Ollama 是否安装。
- Ollama 服务是否正在运行。
- 模型是否已 pull。
- base URL 是否正确。
- API key 是否正确。
- 模型能力是否和配置匹配。

本地 Ollama 默认地址通常是：

```text
http://127.0.0.1:11434
```

### 17.4 SAM 安装失败

可能原因：

- 网络无法下载依赖。
- Python 环境缺少兼容包。
- 权重文件下载失败。
- 磁盘空间不足。

处理：

- 先不用 SAM，手动绘制区域。
- 检查模型管理任务状态。
- 重试安装。
- 使用远程分割服务。

### 17.5 地图或素材丢失

检查：

- `runtime_project/assets/` 是否还在。
- `world.sqlite` 是否来自同一项目。
- 是否只复制了数据库，没有复制 assets。
- 是否误删了用户数据目录。

迁移项目时必须同时复制数据库和素材目录。

### 17.6 打包版和开发版数据不一致

这是正常的，因为默认数据目录不同：

- 开发版：仓库根目录 `runtime_project/`。
- 打包版：用户数据目录 `runtime_project/`。

如果要让两者使用同一份数据，可以通过环境变量 `AGENT_ENGINE_PROJECT_DIR` 指定，但普通用户不建议这么做。

## 18. 快捷检查清单

### 18.1 新用户

- 已启动应用。
- 后端已连接。
- 能看到地图或默认沙盒。
- 能选中 agent。
- 能打开模型管理。
- 能启动或单步模拟。
- 事件日志有新事件。

### 18.2 建图前

- 地图尺寸合适。
- 背景图清晰。
- 已备份旧项目。
- 主要区域类型已规划。
- agent 数量不要一开始太多。

### 18.3 接入 LLM 前

- mock provider 流程已跑通。
- agent 的身份和目标写清楚。
- 动作空间合理。
- 可行走区域和障碍没有明显错误。
- 事件日志正常。

### 18.4 分发前

- 已运行基础测试。
- 已打包 installer 或 portable。
- 已在干净 runtime 目录烟测。
- 已说明未签名风险。
- 已说明本地模型不随包分发。
- 已附带本用户手册。

## 19. 术语表

- Agent：世界中的行动者。
- Item：世界中的物体。
- Region：地图上的功能区域。
- Tick：模拟推进的最小时间单位。
- World snapshot：当前世界状态快照。
- Event：世界中发生的普通事件。
- Decision event：模型决策审计事件。
- Provider：模型服务提供者。
- Mock provider：内置测试模型提供者。
- SAM：图像分割模型，用于从地图背景中切出区域。
- Runtime project：本机运行数据目录。
- Action extension：用户自定义动作扩展。
- Narrative：场景叙事配置和自动旁白。

## 20. 推荐使用方式

如果你是第一次使用，推荐顺序是：

1. 用 portable 打开应用。
2. 先不接真实 LLM，使用默认 mock 行为熟悉界面。
3. 导入一张简单地图。
4. 手动绘制少量区域。
5. 创建 2 到 3 个 agent。
6. 单步推进模拟并查看事件日志。
7. 接入 Ollama 小模型。
8. 调整 agent 身份、目标和对话策略。
9. 再尝试地图生成、SAM 分层和动作扩展。

这样最稳。先让小场景跑顺，再逐步增加复杂度。
